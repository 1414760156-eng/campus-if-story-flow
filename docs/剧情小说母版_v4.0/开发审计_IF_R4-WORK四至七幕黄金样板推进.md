# IF R4-WORK 四至七幕黄金样板推进审计

更新时间：2026-05-26

## 结论

`R4-WORK` 应作为第一条完整支线黄金样板，先从第四幕接到第七幕并完成可试玩验收；其它支线按该样板对齐后，再回头处理母本主轴。这样可以先固定“支线池内自演化”的结构、选择密度、反馈页标准和毕业结算规则，避免各线开发时重新漂移。

当前 `R4-WORK` 并不是从零开始：正式 JSON 已覆盖 P0-B / Act4 / Act5 / Act6 / Act7。但它还不能直接视为“全线样板完成”，因为 Act5 至 Act7 的选择反馈粒度仍低于 Act4 最新样板。

## 主要缺口

1. Act5 / Act6 / Act7 的 `choice_window.directions` 多数仍是 `label + variable_delta`，缺少 Act4 已确认的玩家侧选择后剧情反馈。
2. Act5 需要成为从第四幕工作现实进入第五幕站队 / 亲密 / 项目 / 活动压力的桥，不应只是变量分发。
3. Act6 可以保持双选，但双选后也必须有行动反馈和承接页，不能点完直接进入下一组。
4. Act7 是毕业结算层，选择可以更少，但每个事件仍要让玩家看到毕业温度如何变化，而不是只看到结算标签。
5. 预览器目前主要服务 `R4-WORK Act4`，还没有一条可从 Act4 点到 Act7 的基础试玩样板。

## 黄金样板标准

`R4-WORK` 全线样板完成时，至少满足：

- Act4：保持当前冻结样板，不再细抠表达。
- Act5：5 个桥接节点均有选择前剧情、三方向选择、每方向选择后反馈、进入 Act6 的回声。
- Act6：5 个定型组均有 4 页剧情、双方向选择、每方向选择后反馈、进入 Act7 的结算变量。
- Act7：12 个毕业事件均有 4 页剧情、双方向选择、毕业温度 / 完成度轴变化、结局候选回声。
- 玩家侧文本不得出现系统说明、变量名、路线标签、策划说明或开发记录。
- 全线保持 `POOL-R4-WORK` 锁定，不打开其它路线池。
- 预览器至少能按 Act4 -> Act5 -> Act6 -> Act7 顺序浏览并点选方向，作为后续路线试玩样板。

## 推荐执行顺序

1. 先做 `R4-WORK` Act5 全节点反馈补齐：补 5 个选择窗的 15 个方向选择后剧情反馈。
2. 再做 `R4-WORK` Act6 双选反馈补齐：补 5 个选择窗的 10 个方向选择后剧情反馈。
3. 再做 `R4-WORK` Act7 毕业事件反馈补齐：补 12 个事件的 24 个方向选择后反馈和结局温度说明。
4. 做 `R4-WORK` 4-7 全线一致性审计：前后链、变量、source_docs、完成条件、旧字段残留。
5. 做基础全线试玩预览：先满足读数据、顺序浏览、点选反馈，不做 H5 产品化。
6. 冻结 R4 全线样板后，按同样标准推进其它支线。
7. 所有支线结构对齐后，再回头做母本主轴，避免主轴先写死后支线被迫歪斜。

## 不做事项

- 不继续逐句优化 Act4 表达。
- 不在本阶段重写母本主轴。
- 不把预览器做成正式 H5，只做基础试玩样板。
- 不因为 Act6 / Act7 是双选就省略选择后剧情反馈。
